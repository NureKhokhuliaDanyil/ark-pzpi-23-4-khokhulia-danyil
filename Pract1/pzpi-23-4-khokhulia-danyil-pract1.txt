Харків 2025

1 МЕТА
     Ознайомити студентів з основними рекомендаціями щодо написання чистого, ефективного та підтримуваного коду для різних мов програмування, а також навчити аналізувати та рефакторити код для покращення його якості.

2 ЗАВДАННЯ
	Студент повинен обрати мову програмування, дослідити основні рекомендації для написання якісного коду у цій мові, продемонструвати ці рекомендації на прикладах коду, обґрунтувати кожну рекомендацію і надати детальні пояснення. Для кожного прикладу мають бути наведені фрагменти коду.

3 ОПИС ВИКОНАНОЇ РОБОТИ
     Дотримання єдиного стилю кодування є фундаментом для командної розробки. Уніфікація стилю дозволяє розробникам не витрачати час на розбір візуальної структури ("чому дужка тут, а не там?"), а одразу зосередитися на логіці. Стандартизація спрощує Code Review та зменшує кількість конфліктів при злитті гілок (merge conflicts), оскільки форматування коду залишається стабільним.
     У мові C# діє правило: один файл — один основний публічний тип. Ім'я файлу повинно точно збігатися з ім'ям класу (наприклад, клас OrderController у файлі OrderController.cs). Всередині класу елементи повинні розташовуватися у суворому порядку для швидкої навігації:
1. Константи (const).
2. Приватні поля (private fields).
3. Конструктори.
4. Властивості (public properties).
5. Методи (спочатку публічні, потім приватні допоміжні). Між цими групами елементів обов'язково залишається один порожній рядок для візуального розділення.
     Форматування забезпечує читабельність. Для C# прийняті наступні жорсткі правила:
• Відступи: Використовується 4 пробіли (не табуляція).
• Стиль дужок: Використовується стиль Allman (або BSD style), де відкриваюча фігурна дужка { завжди переноситься на новий рядок. Це стосується класів, методів, циклів та умовних операторів.
• Пробіли: Оператори (=, +, ==, =>) завжди відокремлюються пробілами з обох боків. Після коми у списку параметрів ставиться пробіл. Після ключових слів (if, for, foreach) ставиться пробіл перед дужкою, але після назви методу пробіл не ставиться (MethodName()).
• Логічні блоки: Всередині методів логічні блоки коду відокремлюються одним порожнім рядком. Перед оператором return, якщо він не є єдиним рядком у блоці, також рекомендовано ставити порожній рядок.
     Правильне іменування дозволяє зрозуміти призначення змінної без коментарів. У C# використовуються такі конвенції:
• PascalCase (Перша Літера Велика): Використовується для назв Класів (UserService), Методів (CalculateTotal), Властивостей (FirstName), Констант та Просторів імен.
• camelCase (перша літера мала): Використовується для локальних змінних (userIndex) та параметрів методів (orderId).
• _camelCase (з підкресленням): Стандарт для приватних полів класу (_databaseConnection). Це дозволяє миттєво відрізнити поле класу від локальної змінної.
• Інтерфейси: Завжди починаються з префіксу I (IRepository, ILogger).
• Асинхронні методи: Завжди закінчуються суфіксом Async (SaveDataAsync). Слід уникати угорської нотації (префіксів типу strName, iCount) та однолітерних назв (окрім лічильників i, j у циклах).
     Код повинен бути самодокументованим. Коментарі // використовуються лише для пояснення складної бізнес-логіки ("ЧОМУ це зроблено"), а не синтаксису. Для публічних методів та класів обов'язковою є XML-документація (починається з ///). Вона повинна містити теги:
• <summary> — короткий опис призначення.
• <param name="..."> — опис вхідних параметрів.
• <returns> — опис результату. Це дозволяє середовищу розробки (IDE) показувати підказки при наведенні курсору на метод.
     Ручна перевірка стилю є неефективною. Для автоматизації використовуються:
• EditorConfig: Файл .editorconfig у корені проєкту задає правила (відступи, стиль дужок) для всіх розробників, незалежно від налаштувань їхньої IDE.
• Code Cleanup: Функція Visual Studio для автоматичного форматування файлу при збереженні.
• Статичні аналізатори: Інструменти на кшталт SonarQube або вбудовані Roslyn Analyzers автоматично підсвічують порушення стилю (наприклад, невикористані змінні або порушення іменування) як помилки компіляції.
     
4 ВИСНОВКИ
     У ході виконання роботи було проаналізовано фундаментальні архітектурні принципи проектування програмного забезпечення. Було проаналізовано стандарти оформлення програмного коду на прикладі C#, що підтвердило критичну важливість дотримання єдиного стилю (Coding Conventions) для ефективної командної розробки. Встановлено, що чітка структура, правильне іменування та форматування дозволяють розробникам зосередитися на бізнес-логіці, значно прискорюють процес Code Review та спрощують підтримку проєкту в довгостроковій перспективі.
